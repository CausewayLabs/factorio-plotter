## Manipulation

- [x] (planned: 2026-05-23, completed: 2026-05-23) Drag bubbles (pointer drag → `moveBubble` → solver recompute via microtask).
- [x] (planned: 2026-05-23, completed: 2026-05-23) Rail endpoint editing + whole-rail relocate (select mode, default cursor): `hitTestRailEndpoint` grabs an endpoint within 12px → drag stretches that end, snapped H/V via `orthoSnap` against its neighbour (keeps the orthogonal invariant; the dragged segment can rotate H↔V by dominant delta). Pressing the rail body and dragging translates the whole polyline; a press with no drag (>3px) opens the materials menu on mouseup. RailLayer shows small endpoint handles in select mode for discoverability; a fork's parametric first point gets no handle (not directly draggable). — decision: gesture lives in Canvas via a `railDrag` ref distinguishing click vs drag; reuses the existing `updateRailPoints` store action so the solver recomputes for free.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Rail T-fork: pull a branch from a rail; origin stored as parametric `{parentRailId, t}`, resolved at read time in both solver and hit-test so it slides (clamped) when the parent reshapes.
- [x] (planned: 2026-05-23, completed: 2026-05-23, ~~superseded~~: 2026-05-24) Branch a bus off a bubble's output via `bubbleOrigin: bubbleId` (anchored first point tracking the output port). **Reverted** — anchoring forced sloping geometry on rail-drag and bubble-drag and a "preserve vs. derive length" dilemma. Replaced by the output-connector model below. The `resolveRailPolyline` consolidation (one resolver for fork origins across renderer/hitTest/solver) is kept; only its bubble-origin branch and the `bubbleOrigin` field are removed.
- [x] (planned: 2026-05-24, completed: 2026-05-24) **Emit output onto a bus (output-connector model).** Replaced `bubbleOrigin` with `Bubble.outputTarget: railId | null`. Drag from the output port: drop on a rail → add the bubble's product to that rail's `resourceTypes` + set `outputTarget`; drop on empty space → create a new single-resource rail there + bind. Solver emits a derived **orthogonal** output connector (`bubbleOutputPort` → nearest point on resolved rail via `orthogonalConnector`) that re-routes on any move. Rails stay free orthogonal geometry; downstream feeders tap the product off the bus automatically (reads `resourceTypes`, no matching change). Backed out `bubbleOrigin` across types/geometry/hitTest/RailLayer/Canvas/persistence; `resolveRailPolyline` simplified to `(rail, rails)`. — decision: chose a distinct `OutputConnector` type over tagging `Feeder` with a kind, because feeders are direct 2-point and connectors are orthogonal — conflating them would scatter `if (kind)` branches through routing/rendering. New renderer `OutputConnectorLayer.tsx`. Typechecks clean; in-app smoke still pending.
- [x] (planned: 2026-05-24, completed: 2026-05-24) Output binding cleanup on delete: deleting a bound rail clears `outputTarget` on bubbles pointing at it (`deleteRail` in `scene/store.ts`); deleting a bubble drops its own binding implicitly. — decision: deferred the optional "remove the product from the bus when no other producer remains" — `resourceTypes` is authored, so leaving it lets the user keep the lane intentionally.
- [ ] (planned: 2026-05-23) Polish fork-rail UX: the toolbar hint says "Click on a rail to fork it," but the actual flow requires right-clicking a rail → "Fork Rail" first to set the target. Align the hint/flow (QA note).
- [x] (planned: 2026-05-26, completed: 2026-05-26) **Drag-from-missing-input → create producer bubble.** Mirror of the output-emit-drag. `hitTestUnsatisfiedInputTab` (filters `slot.satisfied === false`) in `editing/hitTest.ts`; `inputFillDrag` ref in `Canvas.tsx` with dashed straight-line preview from port→cursor; mouseup picks first recipe from `getRecipesForProduct(resourceType)` (primary-product-first then id-sorted) and `addBubble({ recipeId, outputBindings: all-null, position: dropPoint })`. Outputs stay unbound — solver wires the consumer via the existing bubble-output-as-feeder-source path; the original consumer's missing-requirement flag clears automatically. — decision: Option 2 (unbound outputs) over auto-binding to a nearby bus, consistent with existing bubble↔bubble feeder behavior; multi-producer disambiguation deferred to a follow-up. Drop on existing bubble/rail explicitly out of scope.

## Multi-select

- [x] (planned: 2026-05-30, completed: 2026-05-30) **Multi-select + group move.** Shift+click a bubble toggles it in/out of `selectedBubbleIds` (Canvas local `useState<Set<string>>`). Click a bubble outside the current selection resets to single-bubble selection. Drag a bubble that is already in the selection → group drag: delta from `dragAnchorWorld` applied to all selected bubbles' original positions (`selectedOrigPositions` ref). Shift+drag on empty canvas → dashed selection-box rect (stroke scaled by `1/viewport.zoom`); mouseup selects all bubbles whose centre falls inside, merged into existing selection. Click (no drag) on empty canvas clears selection. Selected bubbles render a dashed white ring (`BUBBLE_RADIUS + 6`, `stroke: #e0e8ff`, `strokeDasharray: 4,3`, `opacity: 0.7`) in `BubbleLayer`. — decision: selection state lives in Canvas (not scene store) because it doesn't persist and the scene store is already owned by the solver/persistence boundary. `BubbleLayer` received a new `selectedIds: Set<string>` prop.
