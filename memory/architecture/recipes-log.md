# Feature Log: Recipes

*Tracks planned features, open bugs, and completed work for the recipes subsystem.*

## Data Schema & Bundled Set

- [x] (planned: 2026-05-23, completed: 2026-05-23) Bundled source-bus presets (`recipes/sources.ts`): a `SourcePreset { id, label, resourceTypes[] }` seed list. First entry: **Scrap** = the 12 direct Fulgora scrap-recycling outputs (iron gear wheel, solid fuel, concrete, ice, stone, steel plate, battery, copper cable, advanced circuit, processing unit, low density structure, holmium ore) — verified against the Factorio wiki (yields sum to 60%). New colors added for concrete/ice/holmium-ore/scrap.

- [x] (planned: 2026-05-23, completed: 2026-05-23) Define recipe schema: `{ product, variantId, label, inputs: string[], isDefault }`; resource types by name only, no quantities.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Author bundled recipe JSON covering core Factorio intermediates — 29 variants from raw resources through rocket components (circuits, plates, steel, plastic, gears, etc.) with default variants.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Replace the hand-seeded bundled set with a **generated** full base + Space Age catalog (~578 products / 654 variants) via `scripts/generate-bundled-recipes.mjs` from the FactorioLab "spa" dataset. — decision: fixed wrong labels (Advanced Circuit, not "Red Circuit"; Electronic/Processing too) and missing basics (ice, concrete, holmium-ore + all ores/fluids); generator excludes recycling/mining/barreling so raws stay leaves, keeps exotic recipes as non-default variants, re-adds the circuit "flat" abstractions. Source/method promoted to recipes.md.
- [x] (planned: 2026-05-31, completed: 2026-05-31) **Collapse the `steam-165` temperature variant into `steam`.** FactorioLab models steam at two temperatures as distinct `fluids` items — `steam` ("Steam (500°C)", heat-exchanger) and `steam-165` ("Steam (165°C)", boiler) — which surfaced in the catalog as a second steam product with no icon/color, rendered by the app under its producing recipe's label "Boiler : Steam". Fixed both the generator and the committed catalog: (1) `generate-bundled-recipes.mjs` now builds a `fluidTempMap` that collapses any `<fluid>-<temp>` id to its base **only when both variant and base are category `fluids`** (so tiered items like `efficiency-module-2`/`assembling-machine-2` are untouched — the naive "base id exists" rule would wrongly merge those), normalizes recipe in/out through it (label-match still tests the RAW product so "Boiler : Steam" keeps its name), and drops `-<variant>`-suffixed recipe clones that become structural duplicates after normalization; (2) applied the equivalent surgical edit to `bundled.json` directly — boiler product `steam-165`→`steam`, deleted the now-duplicate `coal-liquefaction-steam-165` and `steam-condensation-steam-165` clones. `steam-165` is the ONLY fluid temp-variant in the whole SPA dataset. Result: one `steam` product, three producers (Boiler/Heat Exchanger/Acid Neutralisation). — decision: did NOT ship a full regen because the committed catalog has diverged from generator output (see bug below); applied the generator fix for durability + a scoped hand-edit for the actual catalog.
- [ ] (planned: 2026-05-31) **BUG: generator output has diverged from the committed `bundled.json`.** Running `node scripts/generate-bundled-recipes.mjs` re-adds ~261 research/technology recipes that commit `67c51a2` manually deleted (e.g. `coal-liquefaction-technology-default` and other `*-technology` leaves synthesized by the "ensure every referenced input exists" pass). `isExcluded` only drops *primary*-`-technology` and all-science-pack-input recipes, not `-technology` items pulled in as referenced inputs. So a naive regen balloons the catalog by ~2100 lines and undoes the curation. Fix: make the generator reproduce the research trim (filter `-technology` ids out of inputs and skip synthesizing `-technology` leaves) so regen is a safe, isolated operation again. Until then, the generator's `steam` normalization will not reach the catalog via regen — only via hand-edit.
- [x] (planned: 2026-05-26, completed: 2026-05-26) **Bundle FactorioLab icon sprite sheet + atlas.** Generator now also downloads `icons.webp` (~1.7 MB) and writes `src/recipes/iconAtlas.json` shaped `{ spriteSize: 66, entries: Record<iconId, {x, y}> }` (896 entries). FactorioLab's `data.json` provides a top-level `icons` array of `{ id, position: "-Npx -Mpx", color }` — the generator parses CSS `background-position` strings into non-negative `{x, y}` integers on a fixed 66×66 grid. New module `src/recipes/iconAtlas.ts` exposes static-import `getIconCoords`, `resolveRecipeIconId` (recipe.icon ?? products[0]), `iconsUrl` (Vite `?url` asset), and `SPRITE_SIZE`. BubbleLayer renders the sprite via SVG `<image>` + per-bubble `<clipPath>` (radius = `BUBBLE_RADIUS - 4`); `BUBBLE_RADIUS` grew 36 → 40 to fit a 64-ish icon with padding; recipe name moved to a native `<title>` tooltip; truncated text remains as fallback when an atlas entry is missing. — decision: kept atlas as a separate JSON file (not inlined into `bundled.json`) so the two pipelines can regenerate independently; chose single shared sprite-sheet `<image>` per bubble translated to center the cell over per-bubble data URIs (DOM-light, zoom-crisp).

## Variant Selection & Editor

- [x] (planned: 2026-05-23, completed: 2026-05-23) Per-bubble recipe dropdown (bubble context menu): switch variant updates inputs and triggers solver recompute.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Recipe/product editor UI: create or edit a product and its recipe variants. — decision: mounted at App level, opened via the "Recipes" toolbar button (new products) and the bubble context-menu "Edit Recipe…" (existing product); state held transiently in the editing store.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Palette (ProductPicker) reflects user-authored products via `getAllProductIds()`, so new products are placeable as bubbles.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Recipe Editor left dock: filterable list of all existing products (label, input-count/raw badge, ★ user-authored marker, variant count) — click loads it into the form; "+ New" clears the form for authoring. — decision: a browse pane removes the "am I duplicating an existing recipe?" blindness; kept inside the editor modal as a two-column layout rather than a separate canvas dock.
- [x] (planned: 2026-05-29, completed: 2026-05-29) **Any product can be declared a raw input.** Previously "raw" was only selectable for products that happened to have a catalog recipe with empty inputs (ores/fluids/barrels); intermediates like holmium-plate offered no raw option in the picker or context menu. Added synthetic virtual raw recipes: `recipes/store.ts` exports `makeRawRecipe(productId)` → `{ id: 'raw:<productId>', label: '<Name> (raw)', inputs: [], products: [productId] }`. `getRecipeById` re-synthesizes any `raw:` id not in the catalog (so it round-trips through persistence with no migration), and `getRecipesForProduct` appends a raw option for any product lacking a catalog raw (so the bubble context menu's recipe list always shows it). `ProductPicker` surfaces a "(raw)" row for products matched by name that have no catalog raw. — decision: extended the existing "raw = empty-inputs recipe" model with on-demand virtual recipes rather than adding a per-bubble `isRaw` flag — no schema change, no solver/renderer special-casing (solver/label/icon/color all read `.inputs`/`.products` through `resolveRecipe`), no persistence migration. Contract promoted to recipes.md.

## Product Identity & Matching

- [x] (planned: 2026-05-23, completed: 2026-05-23) Canonical product identity for source-matching: `recipes/normalize.ts` exports `canonicalProductKey` (case-folded, hyphens/underscores/whitespace stripped) + `resolveProductId` (free text → known id, by id or label). The solver matches a bubble's needed input against rail `resourceTypes` and bubble `productId` by canonical key, so "Copper Plate" / "copper_plate" / "copper-plate" are one resource; `getResourceColor` is canonical-keyed too. — decision: a matching aid only — stored ids and labels are untouched (no migration); fixes "feeder finds no source" caused by case/hyphen drift between authored rails and recipe inputs.
- [x] (planned: 2026-05-23, completed: 2026-05-23) Recipe Editor ingredients are now chips added via a forced-match autocomplete (datalist of product labels; Enter/click resolves to a real product id via `resolveProductId`). A no-match draft shows a deliberate "Create <name>" escape (chip flagged `new`) rather than silently adding a typo. — decision: chose autocomplete + explicit-create over free comma-separated text to kill typos that broke source-matching, while preserving the charter's implicit-creation as a conscious act (user-confirmed).
- [x] (planned: 2026-05-30, completed: 2026-05-30) **Filter leveled research recipes from ProductPicker.** Extended the `-technology` suffix filter to also exclude recipes whose inputs are exclusively science packs (`*-science-pack`), catching braking force, mining productivity, train speed, worker robot speed, and inserter capacity upgrade techs. Guard: `r.inputs.length > 0` prevents the every-on-empty-array footgun from dropping raw/leaf recipes. Both conditions together cover all FactorioLab research entries — the suffix filter catches default tech recipes (no inputs); the science-pack input filter catches leveled upgrade techs. — decision: filter applied in `ProductPicker` only, not the store, since it is a display concern; RecipeEditor intentionally keeps all recipes.
- [x] (planned: 2026-05-30, completed: 2026-05-30) **ProductPicker search fixes (three bugs).** (1) Technology recipes (`products` all ending in `-technology`) are now filtered out of `allRecipes` in `ProductPicker` — they are research unlock nodes, not factory producers. (2) Product matching now also compares against `prettify(p)` so typing "iron plate" (space) finds recipes that produce `iron-plate` (hyphen id); previously only the raw id matched the product-path. (3) The "via X" tag now displays `prettify(matchedViaProduct)` ("Iron Plate") instead of the raw id ("iron-plate"). — decision: filter applied in the component, not the store, since technology exclusion is a display concern specific to the picker (RecipeEditor keeps them for user-authored edits). `prettify` already imported by labels.ts for this purpose.
- [x] (planned: 2026-05-28, completed: 2026-05-28) Single-type rail labels now render the decoded material name ("Lithium Plate") instead of the raw kebab id ("lithium-plate"). Extracted the title-casing helper into `recipes/labels.ts` (`prettify`) and replaced the two duplicate copies in `BubbleLayer`/`RecipeEditor`; `railBusLabel` now calls it. — decision: `prettify` (not the product→recipe-label map) is the display source for materials — a rail names a *material*, and the recipe label can diverge from the product name (e.g. "petroleum-gas" produced by "Advanced Oil Processing"). The bundled catalog carries no separate item display names, so title-casing the id is the canonical render.
